## Parent Epic

## Description
AWS上の4ノード環境でコンテンツ操作と権限管理の全機能を検証する。

> **ノード数の前提変更（3 → 4）:** BFT（N ≥ 3f+1, f=1）の最小構成は 4 ノード。検証も 4 ノードクラスタで行い、加えて 1 ノード障害（f=1）下での継続動作を確認する。

## Tasks
- [ ] monas-accountを使ったAuthToken/UCANトークン生成
- [ ] コンテンツ作成（POST /content）→ 他ノードへの同期確認（残り3ノード）
- [ ] コンテンツ更新（PUT /content/:id）→ CRDT同期確認
- [ ] コンテンツ削除（DELETE /content/:id）→ 全4ノードから削除確認
- [ ] 権限付与（POST /content/:id/access/grant）
- [ ] 権限を付与されたユーザーによる操作確認
- [ ] 権限剥奪（POST /content/:id/access/invalidate）→ アクセス不可確認
- [ ] 異なるノード経由でのクロスノード操作確認（4ノード間）
- [ ] **BFT 検証: 1 ノードを停止（f=1）した状態で、残り 3 ノードでコンテンツ CRUD・権限操作が継続できることを確認**
- [ ] **BFT 検証: 停止ノード復帰後に未同期 operation がキャッチアップされることを確認**

## 現状（As-Is）
- 4ノード前提の検証タスクは定義済みだが、各タスクの実行条件・完了条件が明文化されていない。
- 同期確認（CRDT収束、削除反映、復帰後キャッチアップ）の観測方法が未定義。
- 権限操作（grant/invalidate）で想定する成功/失敗レスポンスが未定義。
- BFT検証（1ノード停止中の継続動作）で、停止対象ノード選定や復帰時の確認順序が未定義。
- 実装後の回帰テスト方針（どこまで自動化するか、どこを手動確認するか）が未定義。

## 修正方針（To-Be）
### 1. 検証基準の明文化
- 各タスクに対して、以下を定義する。
  - 前提条件（利用ノード、利用ユーザー、初期データ）
  - 実行操作（API・入力値）
  - 期待結果（HTTPステータス、レスポンス要点、全4ノードでの整合条件）

### 2. ノード間整合性の確認手順を固定
- CRUDごとに「どのノードへ書き込み、どのノードで読み取り確認するか」を固定する。
- CRDT同期は「更新後に全ノードで同一内容を参照できること」を判定基準にする。
- 削除は「全ノードで非存在（404等）となること」を判定基準にする。

### 3. 権限モデル検証の明確化
- grant後は、付与対象ユーザーで許可操作が成功することを確認する。
- invalidate後は、同ユーザーの同操作が拒否されることを確認する。
- 必要に応じて、拒否理由（認可失敗）をレスポンスで確認する。

### 4. BFT停止/復帰シナリオの段階化
- 停止フェーズ: 1ノード停止中に残り3ノードでCRUD/権限操作が成立することを確認する。
- 復帰フェーズ: 停止ノード復帰後に、停止中のoperationが追随反映されることを確認する。
- 可能であれば停止ノードをローテーションし、偏りのない確認を行う。

## 実装計画（段階的）
### Phase 1: 検証仕様の追記（このissueで先に確定）
- issueに「前提条件/期待結果/判定基準」を追記して、実装とテストの受け入れ条件を固定する。

### Phase 2: 必要な実装修正
- 仕様との差分がある箇所のみ最小修正する（認可、同期、障害復帰の順で実施）。
- 1論点ごとにPR可能な粒度で進める（混在修正を避ける）。

### Phase 3: 実装後テスト追加
- API統合テスト:
  - CRUDのクロスノード反映
  - grant/invalidate後の許可/拒否
- 障害系テスト:
  - 1ノード停止中の継続動作
  - 復帰後キャッチアップ
- 追加したテストをCIに組み込み、回帰検知できる状態にする。

## 実装後の確認観点（完了条件）
- 4ノードで状態が収束し、ノード入口に依存しない。
- 権限付与/剥奪が期待どおり有効化/無効化される。
- 1ノード障害時にも機能が継続し、復帰後に欠損なく追随する。

## 進め方の合意（Terraform中心 + テストで成立確認）
- 本対応は、原則としてインフラ定義（Terraform）の4ノード化を中心に進める。
- ただし、Issueの完了判定は「4ノード/BFT要件をテストで確認できること」とする。
- そのため、Terraform差分に加えて、検証手順・テストケース（同期/権限/停止復帰）を段階的に整備する。
- 期待挙動が確認できない場合は、アプリ実装（認可・同期・復旧）の最小修正を追加する。
